Cloud Networking – Explained from Ground Up

This page explains cloud networking the way a senior DevOps / Platform / Cloud engineer should understand it. We start from on-prem reality and move step by step into Azure and AWS.

---

1. Why Networking Exists (Very First Principle)

Networking exists for only three reasons.
  • Connectivity – systems must talk to each other
  • Isolation – systems must NOT all talk to each other
  • Control – traffic must be allowed intentionally

Everything in cloud networking is just a software version of physical networking.

Interview sentence: “Cloud networking is on-prem networking implemented in software.”
---

2. On-Prem Networking (What Actually Existed)

In a data center, networking was very physical.

  • LAN – local network inside a building
  • Ethernet cables
  • Switches – connect servers
  • Routers – connect networks
  • Firewalls – allow or block traffic

DMZ (Demilitarized Zone)

DMZ is a buffer network between internet and internal systems.

Why DMZ exists: Internet traffic must never directly hit databases or internal systems.
If DMZ doesn’t exist, one breach exposes the entire data center.
---

3. Virtualization Changed Hardware, Not Networking

When VMware/Hyper-V came:

  • One physical server → many VMs
  • Physical NIC → virtual NIC
  • Physical switch → virtual switch

But networking rules remained the same.

Key realization: A VM still lives inside a network.
---

4. Cloud Networking Core Concept

VPC / VNet

  • Azure → VNet
  • AWS → VPC

A VPC/VNet is simply your private LAN in the cloud.

Why it exists: You need isolation just like an on-prem LAN.
VPC/VNet is the security and routing boundary.
---

5. Subnets – Why We Divide Networks

Subnets are cloud versions of VLANs.

Public Subnet

  • Has internet route
  • Used for load balancers, gateways

Private Subnet

  • No direct internet access
  • Used for applications, AKS/EKS, databases
Golden rule: Databases never live in public subnets.
---

6. NIC – The Most Ignored Component

NIC (Network Interface Card) is how a VM or node connects to the network.

  • IP address belongs to NIC
  • NSG / Security Group attaches here
Why NIC exists: VM without NIC = server without network cable.
---

7. NSG / Security Group vs Routing Table

NSG / Security Group

  • Firewall
  • Allows or denies traffic

Routing Table

  • Decides where traffic goes
  • Does NOT block traffic
Easy memory: NSG = security guard Routing = GPS
---

8. Internet Gateway vs NAT Gateway

Internet Gateway

  • Inbound + outbound
  • Public resources only

NAT Gateway

  • Outbound only
  • Used by private workloads
Why NAT exists: Private systems need updates but must not be exposed.
---

9. Firewalls – Layered Security

  • NSG / Security Group – VM level
  • Azure Firewall / Network Firewall – central control
  • WAF – application-level attacks
Defense in depth, not one firewall.
---

10. Application Gateway vs Front Door

Application Gateway

  • Regional
  • Inside VNet
  • L7 routing

Front Door

  • Global edge service
  • CDN + TLS + routing
Correct design: Front Door → App Gateway → AKS
---

11. VPN and ExpressRoute

VPN

  • Encrypted tunnel over internet
  • Cheap

ExpressRoute / Direct Connect

  • Private physical connection
  • High bandwidth
  • Expensive
Choose based on cost, latency, compliance.
---

12. Final Interview Summary

Cloud networking is just on-prem networking implemented in software. Public traffic terminates at the edge, passes through WAF and load balancers, reaches private workloads, and accesses data via private endpoints.
