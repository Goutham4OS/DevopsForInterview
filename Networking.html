Cloud Networking – From Physical to Virtual

This page explains WHY cloud networking exists, WHAT each component is, and HOW they work together in Azure and AWS.

1. On-Prem Networking (Ground Reality)

WHY: Everything started here. Understanding this makes cloud networking obvious.
  • LAN – Network inside a building
  • Ethernet – Physical cables
  • Switch – Connects servers
  • Firewall – Controls traffic
  • Router – Connects to other networks / internet
flowchart LR Internet --> Firewall Firewall --> Router Router --> Switch Switch --> Server1 Switch --> Server2
Interview memory: On-prem networking = cables + switches + firewalls + routing.

2. Virtualization Era (VMware, Hyper-V)

WHY: Hardware was expensive → multiple VMs on one server.
  • Physical NIC → Virtual NIC
  • Physical Switch → Virtual Switch
  • VM still needs networking
flowchart LR PhysicalServer --> Hypervisor Hypervisor --> vSwitch vSwitch --> VM1 vSwitch --> VM2
Interview memory: VMs still need NICs and switches – they just became software.

3. Cloud Networking Core (VPC / VNet)

WHY: Cloud needed isolation like a LAN without hardware.
  • Azure: VNet
  • AWS: VPC
flowchart TB VPC["VPC / VNet
(Your Cloud LAN)"]
Interview line: “VPC/VNet is a software-defined LAN.”

4. Subnets (Segmentation)

WHY: Same reason as VLANs → security & control.
  • Public Subnet → internet facing
  • Private Subnet → internal only
flowchart TB VPC --> PublicSubnet["Public Subnet
(DMZ)"] VPC --> PrivateSubnet["Private Subnet
(App / DB)"]
Interview memory: Public = has internet route, Private = no direct internet.

5. NIC (Network Interface Card)

WHY: VM without NIC = server without cable.
  • NIC gives IP
  • NIC attaches VM to subnet
  • Security Groups attach here
flowchart LR VM --> NIC NIC --> Subnet
Key fact: All networking decisions eventually apply at the NIC.

6. Security Groups / NSG (Firewall)

WHY: Cloud needs firewall without hardware.
  • Inbound rules → who can enter
  • Outbound rules → who can leave
flowchart LR Internet --> NSG NSG --> VM
One-liner: NSG decides allowed or denied, routing decides path.

7. Routing Tables

WHY: Traffic must know where to go.
flowchart LR VM --> RouteTable RouteTable --> InternetGateway RouteTable --> NATGateway
Memory: Routing = GPS, NSG = security guard.

8. Internet Gateway & NAT Gateway

WHY: Separate inbound from outbound.
  • Internet Gateway → inbound + outbound
  • NAT Gateway → outbound only
flowchart LR PrivateVM --> NATGateway --> Internet
Golden line: “NAT allows egress without exposing ingress.”

9. Private Endpoint / PrivateLink

WHY: Access PaaS without public internet.
flowchart LR AKS --> PrivateEndpoint --> Database
Clarification: Not VNet-to-VNet. Service is injected into your VNet.

10. VNet / VPC Peering

WHY: Connect networks privately inside cloud.
flowchart LR VNetA <--> VNetB
Rules: Private IP only, no transitive routing by default.

11. VPN (Site-to-Site / Point-to-Site)

WHY: Secure tunnel over internet.
flowchart LR OnPrem --> Internet --> VPNGateway --> VNet
Remember: Uses IPsec + public IP + shared key.

12. ExpressRoute / Direct Connect

WHY: Predictable latency, high bandwidth, private line.
flowchart LR OnPrem --> Telco --> AzureBackbone --> VNet
Bandwidth meaning: How much data can move per second.
